# Milestone v1.0: MVP

**Status:** ✅ SHIPPED 2026-05-16
**Phases:** 1-2
**Total Plans:** 3

## Overview

First working release of `projudi-monitor` — a Playwright-based Node.js CLI that monitors Brazilian court cases on Projudi TJBA, detects new movimentações since the last run, and prints them to the terminal. State persisted locally in JSON files. No login, no CAPTCHA, no parallel requests.

## Phases

### Phase 1: Core Pipeline

**Goal**: A user running `node index.js` gets new court movimentacoes printed to the terminal, with state persisted so the next run only shows truly new entries.
**Mode:** mvp
**Requirements:** MONO-01, CFG-01, CFG-02, SCRP-01, SCRP-02, SCRP-03, PARS-01, PARS-02, PARS-03, STATE-01, STATE-02, STATE-03, DIFF-01, DIFF-02, DIFF-03, DIFF-04, REL-01, REL-02, REL-03, CLI-01, CLI-02

**Plans:**

- [x] 01-01: Walking Skeleton — scaffold project, navigate Projudi, extract movimentacoes, diff, print, save state
- [x] 01-02: Harden Pipeline — multi-process loop, error isolation, exit codes, full CLI flags, empty-result assertion

**Success Criteria (all met):**
1. Running `node index.js` against a real Projudi case number prints any new movimentacoes grouped by process, or produces no output if nothing changed since the last run.
2. Running the script twice in a row without actual court updates produces identical output (idempotent — no false positives on second run).
3. A broken or unreachable case number logs an error for that case and continues processing the remaining cases; the script exits with code 1 only when at least one case failed.
4. Running with `--dry-run` shows the diff but leaves state files unchanged; `--verbose` emits navigation and timing details to stdout.

### Phase 2: Documentation

**Goal**: A user with no prior knowledge of the project can install, configure a process list, run the monitor manually, and set up a scheduled job by following the README alone.
**Mode:** mvp
**Requirements:** DOC-01

**Plans:**

- [x] 02-01: Write README — installation, configuration, running manually, scheduling on Linux and Windows

**Success Criteria (all met):**
1. A user following the README from scratch can install dependencies, populate `config/processes.json`, and execute `node index.js` successfully without consulting external resources.
2. The README provides copy-paste cron (Linux) and Task Scheduler (Windows) instructions that schedule the monitor to run automatically.

## Milestone Summary

**Key Decisions:**

- Navigation uses `formAcessoPublico` (#codigoHash) — not a "Consulta Pública" button click (RESEARCH.md A1 was wrong)
- Section heading is "Eventos do Processo" not "Etapas do Processo" (RESEARCH.md A3 was wrong)
- Result loads in child frame at /projudi/AcessoPublico (frameset architecture)
- Data rows identified by exactly 7 cells; column [0]=nº [1]=description [2]=date [3]=movedBy
- Use process.exitCode (not process.exit()) so async browser.close() in finally completes before Node exits
- Delay inserted between cases only — not after the last one — to avoid unnecessary wait at end of batch
- README written directly from index.js source of truth; all flags, exit codes, and field names verified before documenting

**Issues Resolved:**

- RESEARCH.md had two wrong assumptions (A1: navigation, A3: section heading) — both caught during live DOM inspection and corrected before any commit
- REL-03 error message used accented `movimentação` — changed to ASCII `movimentacao` for reliable pattern matching in tooling
- networkidle wait unreliable on frameset pages — made non-fatal with `.catch(() => {})`, then located frames by URL pattern

**Issues Deferred:**

None — all v1 requirements shipped.

**Technical Debt Incurred:**

- No SIGTERM/SIGINT handlers (HARD-01, v2 requirement)
- No per-execution log files (HARD-03, v2 requirement)
