**# projudi-monitor**

One-line description: CLI that monitors new movimentações on Projudi TJBA and prints them to the terminal.

---

**## Requirements**

- Node.js 18 or higher
- A caseNumber (teor hash) for each process to monitor — the hex string visible in the Projudi public search URL after submitting a case

---

**## Installation**

Steps:
1. Clone or download this repository
2. `cd projudi-monitor`
3. `npm install` — installs Playwright and its Chromium browser (one-time, ~150 MB)
4. `npx playwright install chromium` — only needed if Playwright does not auto-install the browser

---

**## Configuration**

Explain that the process list lives in `config/processes.json`. Show the exact schema:
- `caseNumber` (required) — the teor hash for the case; must be a lowercase hex string of at least 8 characters
- `label` (optional) — human-readable name printed in output

Include a concrete copy-paste example matching the real file format (array with one object, same structure as config/processes.json):

```json
[
  {
    "caseNumber": "1d49c1c15b8c",
    "label": "Processo Teste"
  }
]
```

Note that multiple processes can be listed; the monitor runs them sequentially with a delay between each to avoid rate limiting.

---

**## Running**

**Manual run:**
```
node index.js
```

Explain what happens: the script opens a headless Chromium, navigates Projudi's public search for each case, compares movimentações against the last saved state, and prints only new entries. No output means nothing changed since the last run. State is saved to `state/<caseNumber>.json` (gitignored).

**Flags:**

| Flag | Effect |
|------|--------|
| `--dry-run` | Queries Projudi and shows the diff but does NOT update state files. Safe for testing. |
| `--verbose` | Prints navigation steps, frame URLs, row counts, and timing to stdout. Useful for debugging. |

Flags can be combined: `node index.js --dry-run --verbose`

**Exit codes:**

| Code | Meaning |
|------|---------|
| 0 | All cases processed successfully |
| 1 | At least one case failed (error logged per case; remaining cases still processed) |

---

**## Scheduling**

Explain the philosophy: the script has no built-in scheduler; use the OS scheduler to run it on a recurring interval.

**### Linux — cron**

Open crontab with `crontab -e`. Add a line to run every 30 minutes and append output to a log file:

```
*/30 * * * * cd /absolute/path/to/projudi-monitor && node index.js >> /absolute/path/to/projudi-monitor/monitor.log 2>&1
```

Instructions:
- Replace `/absolute/path/to/projudi-monitor` with the real path on the machine (use `pwd` inside the projudi-monitor folder to get it)
- Change `*/30` to adjust frequency (e.g. `0 */2 * *` for every 2 hours)
- The `>> ... 2>&1` redirects both stdout and stderr to a log file

**### Windows — Task Scheduler**

Step-by-step with exact UI labels (no PowerShell required):

1. Open **Task Scheduler** (search in Start menu)
2. Click **Create Basic Task** in the Actions panel
3. Name: `Projudi Monitor` — click Next
4. Trigger: choose **Daily** (or Weekly) — click Next; set the start time; tick **Repeat task every** 30 minutes for a duration of **1 day** — click Next
5. Action: **Start a program** — click Next
6. Program/script: enter the full path to `node.exe` (find it by running `where node` in Command Prompt, e.g. `C:\Program Files\nodejs\node.exe`)
7. Add arguments: `index.js`
8. Start in: full path to the `projudi-monitor` folder (e.g. `C:\Users\YourName\projudi-monitor`)
9. Click Finish

To capture output to a log file, change step 6 to use `cmd.exe` with arguments: `/c "node index.js >> C:\path\to\monitor.log 2>&1"` and set Start in to the projudi-monitor folder.

---

**## State files**

Explain that state is stored in `state/<caseNumber>.json` (one file per case). These files are gitignored. Deleting a state file causes the next run to treat all existing movimentações as new. This is intentional — re-run with `--dry-run` first to preview what would be printed.

---

**## Troubleshooting**

Short table of common issues:

| Symptom | Likely Cause | Fix |
|---------|--------------|-----|
| `[ERROR] ...: AcessoPublico frame not found` | Projudi site was slow or temporarily down | Wait a few minutes and retry |
| `[ERROR] ...: No movimentacao rows found` | The case has no Eventos do Processo yet, or the teor hash is wrong | Verify the caseNumber in config/processes.json |
| `[ERROR] ...: Invalid caseNumber` | caseNumber is not a lowercase hex string | Copy the teor hash directly from the Projudi URL |
| Script exits with code 1 but some cases print output | At least one case failed; others succeeded | Check `[ERROR]` lines in output |

All flag names, exit codes, field names, and behavior descriptions MUST match the implementation in `index.js` exactly. Do not invent flags or behaviors not present in the source.
